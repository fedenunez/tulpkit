How it works

One brief, a small team, a gated finish

Writes stay single‑threaded — one implementer builds, the others add judgment. The run can't end until the gates clear.

/tulpkit:plan <goal>

Interactively break a goal into a sorted, nested task tree — deciding per piece whether its spec lives in the task or in a project skill. Feeds /make.

/tulpkit:make <text | task>

Run the full staged orchestration — on free text or an existing task (auto‑moves it inprogress → done). Honors the plan's tier/spec hints and freezes the spec.

/tulpkit:add [bug|feature|…] <title>

File a task into a git‑backed folder system (tasks/list + by‑state symlinks); nestable via --parent.

/tulpkit:list [state]

Show a state in detail plus counters for every other state; tasks.ts tree renders the nested backlog.

/tulpkit:remove <slug>

Delete a task's folder + symlink (recoverable from git); --cascade removes a whole subtree.

/tulpkit:help [topic]

Visual ASCII explainer of the whole plugin — make, gates, roles, tasks, plan, specs.

The honest part

Four guardrails, enforced in code

The cautions the evidence is clearest about — built in, not hidden.

01 · integrity

The tester owns & locks tests; the implementer can't edit them. Tampering is refused.

02 · spec gate

Green ≠ done. Sign‑off needs a spec‑conformance attestation — and a locked spec can't be quietly relaxed.

03 · independence

Reviewer runs on a different model than the implementer — optionally an N‑critic panel.

04 · cross‑vendor

Claude builds; if Codex is installed, OpenAI reviews — two vendors checking each other.

Two vendors, checking each other

Claude builds. Codex reviews.

A model flatters its own work, and that bias is strongest within a model family. So the most independent reviewer is a different vendor. tulpkit checks in code whether Codex is installed and logged in — if it is, the review (and optionally the first‑cut tests) run through OpenAI's Codex; if not, it falls back to a different Claude model. Either way, the reviewer is never the author.

Install & setup

Drop it into any repo

It's a normal plugin directory (.claude-plugin/plugin.json + components at root). The scripts run with npx tsx, so the only requirement is Node (tsx is fetched on first run, or npm i -D tsx for speed). Restart or /reload-plugins after changing agents or hooks.

# load locally for a session claude --plugin-dir /path/to/tulpkit # or install via the bundled marketplace (the repo is its own one-entry catalog): /plugin marketplace add /path/to/tulpkit # or: owner/repo for GitHub /plugin validate . # sanity-check plugin.json + marketplace.json /plugin install tulpkit@fedenunez # <plugin>@<marketplace>

Project memory

Plugins don't auto-load CLAUDE.md, so copy examples/CLAUDE.md and examples/LEARNINGS.md into your repo root. They wire the read-before / append-after learnings loop and point Claude at the plugin's commands and skills.

Tasks system

Git-backed, folder-based, nestable

Run at any tasks/ root — repo root for cross-cutting work, a subproject for scoped work (--root).

tasks/list/{type}-{id}-{slug}/ issue.md [attachments] tasks/by-state/{state}/{folder} -> ../../list/{folder} (relative symlink)

type = issue · task · feature (bug = issue + kind: bug) · state = proposed · pending · inprogress · done · rft · rejected. One symlink per task = its state; issue.md frontmatter mirrors it. Nesting is a logical tree via parent/order frontmatter — folders stay flat, so the id/symlink invariants are untouched, and a plan is just the subtree under a root task.

npx tsx scripts/tasks.ts new --as task --title "Add page" [--parent 3 --order 1] [--root packages/web] npx tsx scripts/tasks.ts move --task 7 --to inprogress npx tsx scripts/tasks.ts tree [--task 3] # nested backlog npx tsx scripts/tasks.ts remove --task 3 [--cascade]

Full schema: the tasks skill. Commit the list/ folder and the by-state/ symlink together.

Under the hood

How orchestration works

The main-session orchestrator owns the state machine (scripts/orchestrator.ts): init → detect → validation → phase → tests-lock → spec-lock → test → review → signoff. Subagents (in agents/) do the work and report; the orchestrator records. This keeps writes single-threaded and avoids relying on ${CLAUDE_PLUGIN_ROOT} inside subagent shells.

Four hard exit gates, all deterministic:

  • Validation, adapted to the project. detect inspects the repo; the policy is tdd (test-first; greenfield default), adapt (existing setup), or manual (recorded exception). For tdd/adapt the CLI runs the real test command and checks exit 0. manual is fenced: refused when a runner is detectable, declared up front, and unavailable once a green run exists — so it can't dodge the Stop hook.
  • Test integrity. The tester locks its tests (tests-lock); if the implementer weakened or deleted them, sign-off is refused — capable models game gates they can edit.
  • Spec-conformance. Sign-off requires a spec_conformance attestation; when the spec is frozen (spec-lock), the locked acceptance criteria must also be intact.
  • Auditable sign-off. Checklist with ≥ 4 non-empty items and zero open review issues (optionally an N-critic panel via --min-critics).

Cross-vendor review (optional). If scripts/codex.ts detect finds Codex installed and logged in, the orchestrator routes the review (and optionally first-cut tests) through it — the most independent check, since self-preference bias is strongest within a model family. No Codex? It falls back to the code-reviewer subagent on a different Claude model than the implementer. The Stop hook (scripts/enforce-signoff.ts) blocks the session from ending until the gates clear, with an 8-block safety valve.

⚠ Stop-hook reliability note

Plugin-delivered Stop hooks have historically had a bug where the block halts instead of continuing. If your Claude Code version doesn't continue on the block, copy the hook into your project's .claude/settings.json (this path is reliable):

{ "hooks": { "Stop": [ { "hooks": [ { "type": "command", "command": "npx tsx /abs/path/to/tulpkit/scripts/enforce-signoff.ts" } ] } ] } }

Honest limitations

What's enforced vs. attested

  • Mechanically enforced vs. attested. Four things are enforced by code and can't be talked past: a real exit-0 test run, the hash-based test-integrity lock, the manual fence, and the spec-lock hash. The rest — the spec_conformance attestation and the ≥4-item checklist — are auditable (they leave a reviewable trail), not machine-verified: free-text the reviewer asserts, and a reviewer that lies satisfies them. The orchestrator recording the gates is also the model doing the work, so the hard floor is the mechanical checks, not the paperwork.
  • The gate proves the test command exits 0, not that the tests are strong; under adapt a thin suite is a weak signal. The tester writing real failure/edge cases and the reviewer's coverage check backstop that.
  • The reviewer is an LLM judge (self-preference/position bias), so it runs on a different model than the implementer by default, and on Codex (a different vendor) when available — the strongest mitigation. For high-stakes work, require an N-critic panel and/or human sign-off too.
  • Test integrity is hash-based: it catches edits to locked tests, so the tester must lock the real suite. It can't judge whether unlocked or pre-existing tests are strong — the reviewer's spec-conformance check backstops that.
  • ${CLAUDE_PLUGIN_ROOT} reliably expands in hooks (used by the Stop hook), but its expansion inside slash-command bodies has been flaky in some Claude Code versions. Commands fall back to locating the script; subagents never call the scripts directly (the orchestrator owns the state machine).
  • Symlinks assume a POSIX filesystem; on Windows, enable developer mode or run under WSL.
  • Multi-agent fan-out helps parallel, read-heavy work; it's overkill for small/coupled tasks. Default to fewer agents — that's what the quick tier of /make is for (the 3 gate-critical roles, same gates, no research/architecture ceremony).
