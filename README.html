How it works

One brief, a small team, a gated finish

Writes stay single‑threaded — one implementer builds, the others add judgment. The run can't end until the gates clear.

/tulpkit:make <text | task>

Run the full staged orchestration — on free text, or on an existing task (then auto‑moves it inprogress → done).

/tulpkit:add [bug|feature|…] <title>

File a task into a git‑backed folder system (tasks/list + by‑state symlinks).

/tulpkit:list [state]

Show a state in detail plus counters for every other state.

/tulpkit:remove <slug>

Delete a task's folder + symlink (recoverable from git).

The honest part

Four guardrails, enforced in code

The cautions the evidence is clearest about — built in, not hidden.

01 · integrity

The tester owns & locks tests; the implementer can't edit them. Tampering is refused.

02 · spec gate

Green ≠ done. Sign‑off needs a spec‑conformance attestation.

03 · independence

Reviewer runs on a different model than the implementer.

04 · cross‑vendor

Claude builds; if Codex is installed, OpenAI reviews — two vendors checking each other.

Two vendors, checking each other

Claude builds. Codex reviews.

A model flatters its own work, and that bias is strongest within a model family. So the most independent reviewer is a different vendor. tulpkit checks in code whether Codex is installed and logged in — if it is, the review (and optionally the first‑cut tests) run through OpenAI's Codex; if not, it falls back to a different Claude model. Either way, the reviewer is never the author.
