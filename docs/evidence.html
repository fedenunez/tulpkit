Foundations

The cheap, high‑leverage core

Assemble the brief before any fan‑out (P0)

Biggest lever
skills/orchestrated-delivery · scripts/orchestrator.ts (P0) · task issue.md template

Context assembly is the single highest‑ROI lever on current models, so the orchestrator compiles one canonical brief — specs, conventions, matching learnings, acceptance criteria — before spawning anyone.

Backed by context‑as‑optimization, 89.1% vs 70.7% · MAST: ~42% of failures are spec failures

Verify externally — cite, don't trust memory

Strongly supported
skills/specialist-protocol · agents/code-reviewer · domain-researcher

Facts, APIs, versions and status codes are checked against the authoritative source and cited, never recalled. Self‑correction only helps when grounded in external feedback.

Backed by needs external feedback (DeepMind) · execution‑grounded refinement · industrial LLM review 94–98%

Specs as skills, progressively disclosed

Supported
skills/* · examples/CLAUDE.md

Durable, discoverable specs instead of context re‑typed each run — directly attacking the largest failure category in multi‑agent systems.

Backed by context engineering · MAST spec failures

Learnings memory wired into every run

Supported
examples/LEARNINGS.md · examples/CLAUDE.md (read‑before / append‑after)

Hard‑won lessons are written back and re‑read, so the system stops repeating mistakes — external memory as a first‑class part of context.

Backed by memory in context engineering · external‑memory pattern

Team shape

A small team, pointed right

Single‑threaded writes — one implementer

Supported
agents/implementer (sole writer) · orchestrator owns the state machine

Exactly one agent writes; the rest add judgment. Splitting writes across agents loses context — teams that tried it merged the roles back.

Backed by Cognition: single‑threaded works · context loss across agents

Five core roles, not eight

Right‑sized
agents/ — researcher · architect · implementer · tester · reviewer (+ ux on‑demand)

A small role team is what reaches state‑of‑the‑art; over‑decomposition adds coordination surface that fails. The two fact‑checkers folded into the reviewer's verify remit.

Backed by role team = SOTA 72.2% · over‑decomposition failures

Medium models in the team; opus where it counts

Cost‑effective
agents/* frontmatter — sonnet for makers; opus for architect & reviewer

A well‑structured team of medium models beats one big solo agent; reasoning‑ and independence‑critical roles (architect, reviewer) get the bigger model.

Backed by medium team beats big solo (Agyn)

Functional roles, not persona costumes

Used carefully
skills/specialist-protocol — self‑frame specifically; verifiers run cold

Roles change what an agent does (tools, scope, review). A persona string ("you are a senior X") doesn't lift factual accuracy and can hurt it, so verifiers drop the confident voice.

Backed by personas don't improve accuracy · "expert" prompts can hurt

Guardrails

The honest cautions — enforced, not hidden

The tester owns tests; the implementer can't touch them

Anti‑gaming
agents/tester · agents/implementer · orchestrator tests‑lock / tests‑verify

Capable models game gates they can edit — rewriting or deleting tests to go green. The tester locks the suite; a hash check at sign‑off refuses if locked tests changed.

Backed by ImpossibleBench: agents game tests · verifier gaming scales with RL

Green ≠ done — a spec‑conformance attestation

Necessary ≠ sufficient
scripts/orchestrator.ts (signoff gate) · signoff-checklist.md

Passing tests can mean weak tests. Sign‑off is refused unless the reviewer attests the implementation meets the acceptance criteria, not merely that the suite is green.

Backed by Goodhart / reward hacking

The reviewer is never the author

De‑bias
agents/code-reviewer — opus, ≠ the sonnet implementer

A model flatters its own work. Self‑preference is confirmed on 2026 frontier judges, so the reviewer always runs on a different model than the implementer.

Backed by self‑preference persists on frontier · position bias

Cross‑vendor review with Codex, when available

Strongest independence
scripts/codex.ts (detect → review/test) · skill conditionally summons it

Claude builds the code; self‑preference is strongest within a model family, so if Codex is installed and logged in (checked by code), the review runs through OpenAI — a different vendor — otherwise it falls back to a different Claude model.

Backed by no judge uniformly reliable (2026)

No ungrounded self‑review loops

Avoided by design
skills/specialist-protocol §5 — improve from feedback, not re‑reading yourself

"Think again" without an external signal doesn't reliably help and can flip right answers to wrong, so every loop is anchored to tests, docs, or the diff.

Backed by when self‑correction works (survey) · the dark side

Validation

Executable, adaptive, honest

The gate runs the real tests — "pass" can't be a claim

Executable
scripts/orchestrator.ts test (spawns the command, checks exit 0)

Execution feedback is the verifier that makes the loop work, so the state machine runs the command itself instead of trusting a "tests pass" assertion.

Backed by execution feedback fixes code · grounded refinement

Adaptive TDD — test‑first by default, not forced

Adaptive
orchestrator detect / validation --mode tdd|adapt|manual

TDD improves quality but costs 15–35% more time, so the policy adapts to the project rather than mandating test‑first everywhere.

Backed by TDD meta‑analysis: quality up, slower

The Stop hook enforces the process — knowingly, not the truth

Process guard
scripts/enforce-signoff.ts (Stop hook, 8‑block safety valve)

Forcing the loop to run a check before stopping attacks premature termination — but we treat it as a process guard, paired with integrity + spec gates, because enforcing a gameable gate isn't proof of correctness.

Backed by premature‑termination failures · why process ≠ truth

Planning & specs

Plan small, freeze the spec, verify in numbers

Added Jun 2026 — grounded while fact‑checking the "Karpathy method" video (see stop-prompting-analysis.md). Each claim was traced to a primary paper; verdicts are deliberately skeptical.

Plan into the smallest independently‑verifiable increments

Strongly supported
commands/plan.md · skills/planning · tasks.ts (nested tree)

The strongest‑supported idea in the whole method: decomposing a goal into small, ordered, individually‑checkable steps beats one big spec. /tulpkit:plan produces a sorted nested task tree; each leaf is one gated /make run.

Backed by least‑to‑most: 99% vs 16% · chain‑of‑thought

Spec‑lock — freeze the acceptance criteria

Anti‑gaming
scripts/orchestrator.ts spec‑lock / spec‑verify · signoff Gate 3 (hardened)

Just as a model can edit tests to go green, it can quietly relax the spec to "conform". Once criteria are agreed, spec‑lock hashes them; a change by sign‑off is refused — conformance only counts against the frozen spec.

Backed by process supervision > outcome · Goodhart / reward hacking

Specs live as skills — discovery is the retrieval layer

Supported
skills/planning · project .claude/skills/<name> (spec: skill:<name>)

Design‑affecting knowledge becomes a project skill. Claude Code already auto‑discovers and progressively loads skills, so the project's skills are the searchable knowledge base — external knowledge demonstrably lifts factual accuracy, and we get it without a bespoke retrieval engine.

Backed by RAG: more factual generations

N‑critic verification panel (opt‑in)

De‑bias
scripts/orchestrator.ts review ‑‑critics N · signoff ‑‑min‑critics N

For high‑stakes work, several independent critics (varied model/vendor, each told to refute) vote — recorded and required at sign‑off. Redundant independent verification catches what one reviewer misses, and a distinct critic sidesteps self‑preference.

Backed by multi‑agent debate · judges favor their own output
